# 从“拆”开始

很多教材会这样开始一章：分治是一种算法设计范式，它将一个复杂的问题分解成两个或更多的相同或相似的子问题，直到最后子问题可以简单地直接求解……

但这不是我们学习的方式。没有人是通过背诵定义来理解思想的。思想是在解决问题的困境中，被迫诞生的。

准备好了吗？我们的第一个问题简单得可笑。

## 1 最大值：一个不需要“拆”的问题
**问题**：给你一个数组：`A = [5, 2, 6, 1, 3]`。请找出其中的最大值。

几乎所有人脑中闪过的第一个，也是唯一一个念头是：遍历。
```python
max_val = A[0]
for i in range(1, len(A)):
    if A[i] > max_val:
        max_val = A[i]
```
我们需要查看每一个元素，并与当前最大值比较一次。对于 `n` 个元素，我们进行了 `n-1` 次比较。

你可能会想：能不能更快？比如，把数组拆成两半，分别找最大值，然后再比较这两个最大值？

试试看。拆成两半需要一次比较，但找左边最大值需要 `(n/2)-1` 次比较，找右边最大值也需要 `(n/2)-1` 次比较。总次数是 `1 + (n/2)-1 + (n/2)-1 = n-1` 次，和遍历一样多。不仅如此，拆分会带来额外的开销，比如管理子数组的边界。在这个问题里，“拆”没有带来任何好处。

**启示**：并非所有问题都能从“拆分”中获益。当解决问题的必要信息无法被“复用”或“规避”时，线性遍历就是最优解。

这给我们提了个醒：“拆”本身不是目的，它必须能带来效率上的提升。那么，什么样的问题，“拆”了会更快呢？我们把问题稍稍改变一下。

## 2 最大值与最小值：信息的复用
**新问题**：给你同一个数组：`A = [5, 2, 6, 1, 3]`。请同时找出其中的最大值和最小值。

**最直接的方法**：我们依然可以遍历：
1. 初始化 `max_val = min_val = A[0]`
2. 对于每个后续元素 `x`：
    - 如果 `x > max_val`，则更新 `max_val`
    - 如果 `x < min_val`，则更新 `min_val`

对于每个元素，我们进行了2次比较。`n` 个元素总共是 `2(n-1)` 次比较。

你可能会觉得：看起来必须对每个元素做两次比较，这似乎无法避免……真的无法避免吗？

让我们看看第一个元素 `5` 和第二个元素 `2`。我们比较它们一次：`5 > 2`。这次比较告诉我们，`5` 可能是最大值，`2` 可能是最小值。在这次比较中，关于“最大值”和“最小值”的信息是同时产生的。我们能不能利用这一点来设计方法？

**一个“奇怪”的策略**：
1. 将元素两两分组。（如果元素个数是奇数，最后一个元素自成一组，同时作为该组的最大值和最小值）。
    - 例如：`(5,2)`, `(6,1)`, `(3)`
2. 在每一组内部进行比较，区分出组内较大者和组内较小者。
    - `(5,2) ->` 较大=`5`，较小=`2`
    - `(6,1) ->` 较大=`6`，较小=`1`
    - `(3) ->` 较大=`3`，较小=`3`
3. 现在，全局最大值一定在所有“组内较大者”中产生。我们在这 `⌈n/2⌉` 个候选者中寻找最大值。
4. 全局最小值一定在所有“组内较小者”中产生。我们在这 `⌈n/2⌉` 个候选者中寻找最小值。

**复杂度分析**：
- 组内比较次数：`⌊n/2⌋` 次。
- 在 `⌈n/2⌉` 个较大者中找最大值：`⌈n/2⌉ - 1` 次比较。
- 在 `⌈n/2⌉` 个较小者中找最小值：`⌈n/2⌉ - 1` 次比较。
- 总比较次数：`⌊n/2⌋ + 2(⌈n/2⌉ - 1)`。

当 `n` 为偶数时，总次数为 `n/2 + 2*(n/2 - 1) = 3n/2 - 2`。
当 `n` 为奇数时，总次数约为 `3n/2 - 2`。

这比 `2n-2` 次要少！当 `n` 很大时，我们节省了大约 25% 的比较次数。

**启示**：当我们试图解决一个复合问题（如同时求最大最小值）时，一次聪明的、局部的操作（如两两比较）可以产生用于解决多个子目标的信息。先进行局部处理，再合并结果，可能比全局线性扫描更高效。

我们隐约摸到了一条路的边缘：先局部解决，再合并。但它的威力似乎仅限于这种“信息复用”的小技巧。它能用来解决真正复杂的问题吗？让我们进入一个更困难的领域。

## 3 逆序对
**新问题**：给定数组：`A = [3, 1, 4, 2]`。定义：如果下标 `i < j`，但值 `A[i] > A[j]`，则 `(i, j)` 构成一个逆序对。请问数组中总共有多少个逆序对？

**例子**：在 `[3,1,4,2]` 中：
- `(0,1): 3 > 1` ✔
- `(0,3): 3 > 2` ✔
- `(2,3): 4 > 2` ✔
共有 3 个逆序对。

**最直接的方法（暴力枚举）**：
```python
count = 0
for i in range(0, n-1):
    for j in range(i+1, n):
        if A[i] > A[j]:
            count += 1
return count
```
复杂度是 `O(n²)`。如果 `n=100,000`，我们需要进行约100亿次比较，这在竞赛中是不可接受的。

你可能会想：必须比较每一对 `(i, j)` 吗？看起来是的，因为任何一对都可能构成逆序对。但 `O(n²)` 太慢了。我们能不能不明确检查所有对，而是通过某种方式“批量”计算它们？

**关键观察：拆开看看**：让我们把数组从中间切开：`[3, 1] | [4, 2]`。现在，所有的逆序对会出现在哪里？
1. 左边内部：`[3,1]` 中有一个逆序对 `(3,1)`。
2. 右边内部：`[4,2]` 中有一个逆序对 `(4,2)`。
3. 跨越左右：左边元素与右边元素构成的逆序对。例如 `(3,2)`。

**一个惊人的递归结构**：如果我们用 `count(A)` 表示数组 `A` 的逆序对总数，那么：
`count([3,1,4,2]) = count([3,1]) + count([4,2]) + count_cross([3,1], [4,2])`
其中 `count_cross(L, R)` 是计算所有左元素大于右元素的对数。

**希望出现了**：`count([3,1])` 和 `count([4,2])` 不就是原问题的缩小版吗？我们可以用同样的方法递归计算！现在我们最大的挑战，变成了如何高效计算 `count_cross(L, R)`。

## 4 跨越逆序对的计算
**问题**：假设我们已经递归计算好了左右两半的逆序对数量，并且——这是一个至关重要的假设——我们在递归过程中顺便将左右两半各自排好了序。即：`L = [1, 3]`, `R = [2, 4]`。现在，如何计算有多少对 `(l, r)`，其中 `l` 来自 `L`，`r` 来自 `R`，且满足 `l > r`？

**错误的思路**：遍历 `L` 中每个元素，在 `R` 中顺序查找有多少个比它小的。这依然是 `O(n²)` 的。

**关键观察：利用有序性**：因为 `L` 和 `R` 都是有序的，我们可以利用归并的过程一次性数出来！

让我们模拟同时遍历两个有序数组的过程，就像合并它们一样：
```
初始化：i 指向 L 开头，j 指向 R 开头，count = 0
结果数组 res 为空
while i < len(L) and j < len(R):
    如果 L[i] <= R[j]:
        将 L[i] 放入 res，i++
    否则 (L[i] > R[j]):
        将 R[j] 放入 res，然后 count += (len(L) - i)，j++
最后将剩余元素放入 res
```

**为什么可以这样**？当 `L[i] > R[j]` 时，由于 `L` 是升序的，`L[i]` 及 `L` 中 `i` 之后的所有元素都大于 `R[j]`。因此，对于当前的 `R[j]`，逆序对数量为 `len(L) - i`。

**过程演示** (`L=[1,3]`, `R=[2,4]`)：
1. 比较 `L[0]=1` 和 `R[0]=2`。`1 <= 2`，不是逆序。将 `1` 放入结果，`i++`。结果 `[1]`。
2. 比较 `L[1]=3` 和 `R[0]=2`。`3 > 2`，是逆序。此时 `len(L)-i = 2-1 = 1`，说明左半部分有1个元素（即 `3`）大于 `2`。`count += 1`。将 `2` 放入结果，`j++`。结果 `[1,2]`。
3. 比较 `L[1]=3` 和 `R[1]=4`。`3 <= 4`，不是逆序。将 `3` 放入结果，`i++`。结果 `[1,2,3]`。
4. 将剩余的 `4` 放入结果。结果 `[1,2,3,4]`。

跨越逆序对数量 = 1。加上左内部(1)和右内部(1)，总数为3，正确。

**算法**：
1. **划分 (Divide)**：将数组平分为左右两半。
2. **征服 (Conquer)**：递归计算左半部分的逆序对数，并返回排序好的左半部分。
3. **征服 (Conquer)**：递归计算右半部分的逆序对数，并返回排序好的右半部分。
4. **合并 (Combine)**：归并左右两个已排序数组。在归并过程中，当从右半部分取元素时，根据上述规则累加跨越逆序对的数量。最终返回排序好的整个数组，以及总逆序对数。

**意外收获**：你可能会惊讶，这个算法在递归计算逆序对的同时，居然把数组排好序了？是的。我们为了解决逆序对问题，被迫采用了“先排序子数组，再高效合并计数”的策略。而这个策略本身，就是一个完整的、高效的排序算法——**归并排序 (Merge Sort)**。

**复杂度分析**：在每一层递归，我们需要线性时间 `O(n)` 来合并两个已排序的子数组并计数。递归树共有 `log₂n` 层。因此总时间复杂度为 `O(n log n)`。我们从 `O(n²)` 的困境中逃脱了出来。

**核心观察**：对于某些问题（如逆序对），其解可以由子问题的解与一个“合并代价”组合而成。如果这个“合并代价”可以控制在 `O(n)` 甚至更低，并且子问题规模能显著减小（如减半），那么递归就能将复杂度从多项式级别降至 `O(n log n)` 级别。“排序”在这里不是目的，而是高效合并子问题答案的必备工具。

## 5 快速幂：改变问题的结构
**新问题**：计算 `a` 的 `n` 次方，即 `a^n`，其中 `n` 是一个非常大的非负整数。

**最直接的方法**：
```python
result = 1
for i in range(n):
    result *= a
```
复杂度 `O(n)`。如果 `n` 是10亿，这太慢了。

**困境**：乘法必须做 `n` 次吗？看起来是的，因为 `a` 要乘以自己 `n` 次。

让我们换个角度看。`a^8` 是什么？是 `a*a*a*a*a*a*a*a`，需要7次乘法。但它也等于 `(a^4)^2`。如果我们先算出 `a^4`，再将其平方，只需要（计算 `a^4` 的代价）+ 1 次乘法。

**关键观察：指数的递归结构**：`a^n` 可以基于 `a^(n/2)` 来构建！
- 如果 `n` 是偶数：`a^n = (a^(n/2))^2`
- 如果 `n` 是奇数：`a^n = (a^(n/2))^2 * a` （这里 `n/2` 指整数除法，即 `⌊n/2⌋`）

**算法浮现**：
```python
def fast_pow(a, n):
    if n == 0:
        return 1
    half = fast_pow(a, n // 2)  # 递归计算子问题
    if n % 2 == 0:
        return half * half      # 合并：一次乘法
    else:
        return half * half * a  # 合并：两次乘法
```

**复杂度分析**：每一次递归调用，`n` 的规模减半。递归树深度为 `O(log n)`。每一层只需要常数次（1或2次）乘法。因此总复杂度为 `O(log n)`。这是一个质的飞跃。

**核心观察**：分治不一定是对物理序列（如数组）的拆分。它可以是对问题规模本身的递归分解。通过将指数 `n` 分解为 `n/2` 的子问题，并将子问题的解以低廉的代价（平方）合并，我们彻底改变了问题的计算结构，从线性链变成了对数深度的递归树。

## 6 分治：思想的浮现
通过以上几个问题的挣扎、尝试和发现，我们现在可以自己来总结，究竟什么是“分治”，以及它为什么有效。

**分治算法的通用模式**：
1. **分解 (Divide)**：将原问题分解成若干个规模较小、形式相同的独立子问题。
2. **解决 (Conquer)**：递归地解决这些子问题。如果子问题足够小，则直接求解。
3. **合并 (Combine)**：将子问题的解合并，形成原问题的解。

**分治有效的关键**：
- **子问题独立性**：子问题之间不应有复杂的交叉影响（如逆序对中的跨越计数被抽离到合并阶段处理）。
- **合并的高效性**：合并步骤的代价必须显著小于直接解决原问题的代价。这是降低复杂度的核心。
    - 归并排序：合并两个有序数组，代价 `O(n)`。
    - 逆序对：在合并过程中计数，代价 `O(n)`。
    - 快速幂：将子问题的解平方，代价 `O(1)`。
- **子问题规模的缩减**：子问题的规模必须明显小于原问题（通常是常数比例，如1/2），才能实现递归层数的对数化，从而降低复杂度。

**关于“更快”**：分治并不总是更快。汉诺塔问题（`T(n)=2T(n-1)+1`）本身具有指数级的最优解。分治在这里清晰地揭示了问题的本质结构，但并未降低其固有的复杂度。分治是一种揭示并利用问题内在结构的思维方式，其效果取决于结构本身。

**回到最初的疑问**：为什么“把问题拆开”有时会更快？现在我们可以回答：当一个问题可以被拆解成若干独立的、规模更小的相同问题，并且这些子问题的解能够以低于原问题复杂度的代价被重新组合时，“拆开”就能通过将大量工作下放到更小规模的递归调用中，并通过对数深度的递归树来摊薄合并的代价，从而实现整体效率的提升。

这，就是分治。

## 7 本章遗留的思考题
1. **快速排序**也是一个经典的分治算法。它的“划分”步骤是选择一个主元，将数组分为“小于主元”和“大于主元”两部分。请分析，它的“合并”步骤是什么？为什么在最坏情况下它的复杂度会退化到 `O(n²)`？
2. **最近点对问题**：在二维平面上给出 `n` 个点，找出距离最近的一对点。一个暴力解法是 `O(n²)`。如何设计一个基于分治的 `O(n log n)` 算法？关键挑战在于如何高效地合并左右两半的解，即处理“一个点在左半区，一个点在右半区”的情况。
